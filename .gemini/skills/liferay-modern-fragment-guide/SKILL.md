# Modern Liferay Fragment Guide (Domain Specific)

## 1. Overview
- **[Most Important Core Skill]**: This skill is a **core skill** in Liferay DXP fragment development. You are obligated to **"fully, without omission" address** all rules listed here, especially the [Strict Compliance] items, without any exceptions.
- This skill defines implementation guidelines specifically for the "Fragment" feature of Liferay DXP.
- **Directory Structure**: All fragments must be managed by folder under **`src/fragments/`**.
- **Naming Convention**: You must attach a "unique and easily identifiable prefix" to fragment names and folder names to avoid collisions across projects.
- For design standards, common CSS/JS, and mobile-first principles, **be sure to refer to `modern-ux-core-protocol`.**
- When navigation is required in headers, etc., use `lfr-drop-zone` without holding internal logic in the fragment.
- For the placed navigation widget, **always apply the template generated by the `liferay-nav-template-generator` skill.**
- The display language will be English.
- [Strict Compliance] The content described in this skill already reflects the latest definitions of Liferay DXP, so treat it as the latest. Please strictly adhere to it.
- Use images proactively to increase visual impact.

## 2. HTML: Liferay Specific Tags

### LFR-Editable
Defines areas editable by the user in the Page Builder.
- **Text:** `<h2 data-lfr-editable-id="my-title" data-lfr-editable-type="text">Title</h2>`
- **Image:** `<img data-lfr-editable-id="my-image" data-lfr-editable-type="image" src="..." />`
- **Link:** `<a data-lfr-editable-id="my-link" data-lfr-editable-type="link" href="#">Link Display Text</a>`
- **Span:** `<span data-lfr-editable-id="my-span" data-lfr-editable-type="text">Span Text</span>`

> [!CAUTION]
> **[Strict Compliance] Prohibition of HTML Inclusion and Nesting (Wrapping) in Editable Areas**
> Ensure that only plain text is included inside tags with `data-lfr-editable` (to prevent garbled text and tag escaping).
> It is strictly prohibited to include HTML tags (`<span>`, `<i>`, `<b>`, `<strong>`, `<em>`, etc.) inside.
> If you want to include icons (`<i>`) or badges (`<span>`), place them "outside (e.g., just before)" the editable tag.
> **Furthermore, it is strictly prohibited to nest editable elements together (wrap with another editable element).** Place them independently, or if wrapping like a link is necessary, use standard HTML tags (non-editable).
> **NG (HTML inclusion)**: `<a data-lfr-editable-id="l1" data-lfr-editable-type="link"><i class="fa-solid fa-clock"></i> Link</a>`
> **OK (HTML externalization)**: `<div><i class="fa-solid fa-clock"></i> <a data-lfr-editable-id="l1" data-lfr-editable-type="link">Link</a></div>`
> **NG (Editable nesting)**:
> ```html
> <a data-lfr-editable-id="logo-link" data-lfr-editable-type="link" href="/">
>     <img data-lfr-editable-id="logo-image" data-lfr-editable-type="image" src="..." />
> </a>
> ```

## 3. JavaScript: Fragment Environment
Fragment-specific JS context and data integration.

### Fragment Context
- **fragmentElement:** Directly references the root element (DOM) of the fragment. Use like `fragmentElement.querySelector('.class')`.
- **Mapping:** Utilize the fragment mapping feature to beautifully pour in Liferay Object or Web Content data.

### Data Logic
- When dynamically displaying data using an API, follow the Async/Await pattern and error handling defined in `modern-ux-core-protocol`.

> [!CAUTION]
> **[Strict Compliance] Prohibition of Escaping Backticks During JS Code Generation**
> When generating scripts (JavaScript), do not escape backticks (`) or variable expansions (${}) used in template literals with backslashes (\) (e.g., \` or \${ is prohibited). This causes a JS syntax error. Always output unescaped backticks (`) and variable expansions (${}).

### 4. Dynamic Menu Integration
When integrating a "dynamic site menu" into a fragment, follow these rules.

#### 4.1 Utilization of Dropzone (Fragment Side)
When a header or navigation is needed, define a dedicated `lfr-drop-zone` to place widgets, without holding the internal logic in the fragment.
For convenience of placement and to secure the layout, it is mandatory to also specify **`min-width: 300px;`** in CSS for the `lfr-drop-zone`.

> [!IMPORTANT]
> **Rendered Characteristics**
> When a widget is placed inside an `lfr-drop-zone`, multiple wrapper `div`s like `lfr-layout-structure-item-...` and `portlet-boundary` are automatically generated by Liferay.
> **[Strict Compliance] CSS/JS Design Caveats**:
> 1.  **Avoidance of Child Element Selector (`>`)**: If written like `lfr-chair-header__nav > .ln-menu` in the fragment's CSS, the style will not be applied due to the intervention of the auto-generated wrappers. Always use descendant selectors (` `).
> 2.  **Utilization of Event Delegation**: When accessing elements inside the menu with JS, `fragmentElement.querySelector('.ln-menu')` might not be retrievable immediately after placement. Use event delegation to the parent element (fragment root) to control dynamically added menu elements.
> 3.  **Hierarchy of Mobile Drawer**: If an `lfr-drop-zone` is included inside a mobile drawer (`.lfr-chair-header__nav`), you can achieve a drawer without compromising Liferay's widget placement function by removing `display: none` from the drawer itself and keeping it on standby outside the screen with `transform`, etc.

```html
<div class="header-navigation">
    <lfr-drop-zone></lfr-drop-zone>
</div>
```
#### 4.2 Standard Mobile Navigation Implementation Pattern (Mandatory)
Hamburger menus and drawers for mobile display must "always" be implemented with the following pattern.

1.  **Class Control in JS**: Toggle a class to prevent scrolling on `document.body` and add `.mobile-menu-open` to the header when `mobileToggle` is clicked.
2.  **Implementation of Mobile Submenu (Accordion)**:
    *   If the navigation template placed in the `lfr-drop-zone` has multiple hierarchies, it must **always be expanded in an accordion format** on smartphone display.
    *   Triggered by tapping a parent menu (without a link) or an arrow icon (`.ln-menu__indicator`), add a class like `.mobile-open` to display or hide the submenu.
    *   Include visual feedback by rotating the arrow icon (180 degrees).
3.  **Drawer Placement in CSS**:
...
   - `.header-navigation` (Wrapper) uses `position: fixed` or `sticky` on mobile.
   - Place immediately below the header with `top: var(--header-height);`.
   - Prevent transparency with `background: #ffffff !important;`.
   - Smooth appearance with `transition` (Slide or Fade).
3. **Menu Item Layout**:
   - Change `.ln-menu` to `flex-direction: column;` on mobile.
   - For dropdowns (`.ln-menu__child-list`), consider clicking or always expanded (`position: static;`) instead of hovering.

#### 4.3 Skill Integration: liferay-nav-template-generator [Strict Compliance]
When generating a template (.ftl) to be applied to a navigation menu widget placed in an `lfr-drop-zone`, **always use the `liferay-nav-template-generator` skill.**

#### 4.4 Handling of Icons (FontAwesome) (Mandatory)
From Liferay 7.3 onwards, FontAwesome is no longer included by default. If using `<i>` tags (FontAwesome) within fragments, strictly observe the following rules.

1. **Library Introduction**: Always add a FontAwesome CSS link from an external CDN (e.g., Cloudflare) at the beginning of the fragment's `index.html`.
   ```html
   <link rel="stylesheet" href="https://cdnjs.cloudflare.com/ajax/libs/font-awesome/6.4.0/css/all.min.css" />
   ```
2. **Consistency**: Standardize the version of FontAwesome used in the project.

### Design Guidelines
- **Separation of Structure**: Design the fragment to handle the "framework (Container/Dropzone)" and the generated navigation template to handle the "internal logic (Menu list/Link logic)" of the navigation menu.
- **Handling Images**: We recommend using many images to enhance visual impact. Use high-quality images and apply `object-fit: cover;` as necessary.
- **Style Consistency**: Design class names so that the fragment's CSS variables (Clay/Bootstrap compliant) can be inherited on the navigation template side.
- **Dropdown Control**: When embedding navigation, always specify `position: relative` for the parent element, `position: absolute` for the child list (`.ln-menu__child-list`), and describe hover control (`opacity/visibility`) for desktop display.
- **Opacity of Mobile Drawer**: For menu expansion (like `.mobile-menu-open`) in mobile display, always specify an opaque background (e.g., `background: #ffffff !important`) so that the back does not become transparent.
- **Height Management and Following**: When using `sticky` in a header fragment, dynamically update variables like `--header-height` depending on the scroll state (e.g., `is-shrunk`) so that the drawer (mobile menu, etc.) follows correctly.
- **Ensuring Opacity**: To prevent the back of the mobile drawer from becoming transparent, specify `!important` for the background color, and eliminate the influence of `backdrop-filter`, etc., of the parent element (header).

### 5. Fragment Configuration & Configuration Template
Configuration definitions for fragments and dynamic control of templates using them.

> [!CAUTION]
> **[Strict Compliance] Strict Formatting Required**
> The syntax, field definitions of `configuration.json`, and reference syntax on the HTML side (`${configuration.fieldName}`) require an extremely strict format due to Liferay DXP specifications. Not a single character mistake is allowed.
> Since a writing mistake can cause fragment loading failures or unexpected behavior, you must "absolutely" follow the definitions from 5.1 to 5.4 below, and thoroughly verify consistency when making changes.

#### 5.1 Configuration Definition (configuration.json)
Describe the settings in `configuration.json` in the fragment folder. It mainly consists of `fieldSets` and `fields`.

```json
{
  "fieldSets": [
    {
      "label": "Style Settings",
      "fields": [
        {
          "name": "numberOfFeatures",
          "label": "Number Of Features",
          "description": "number-of-features",
          "type": "select",
          "dataType": "int",
          "typeOptions": {
              "validValues": [
                {"value": "1"},
                {"value": "2"},
                {"value": "3"}
              ]
          },
          "defaultValue": "3"
        },
        {
          "name": "showButton",
          "label": "Show Button",
          "type": "checkbox",
          "defaultValue": true
        }
      ]
    }
  ]
}
```

#### 5.2 Reference Method on HTML Side (index.html)
Use the syntax `${configuration.fieldName}` in HTML to retrieve the value defined in JSON.

```html
<div class="container bg-${configuration.backgroundColor}">
    <h2 data-lfr-editable-id="title" data-lfr-editable-type="text">
        Title
    </h2>
</div>
<div style="display: [#if configuration.showTitleButton??]'block'[#else]'none'[/#if];">
```

#### 5.3 Field Type List

> [!CAUTION]
> **[Strict Compliance] Ensure that only the following definitions are used**

| type | dataType | Purpose |
| :--- | :--- | :--- |
| text | string | Any text input |
| select | string | Dropdown selection (options required) |
| checkbox | boolean | ON/OFF toggle |
| colorPicker | string | Get color code (e.g., #FFFFFF) |
| number | number | Numerical input (slider also possible) |

#### 5.4 Development Tips
- **Name Matching**: The `name` in `configuration.json` and `configuration.name` in HTML must completely match.
- **Importance of Default Values**: If `defaultValue` is not set, it may become an unintended display (empty state) immediately after placement.
- **Reflection to Preview**: Changing values from the "Settings" tab on the right side of the fragment editor reflects it in the HTML in real-time.

## 7. Metadata & Visuals (fragment.json) [Strict Compliance]

To ensure the manageability and visibility of fragments, always include the following settings.

### 7.1 Icon and Path Settings (fragment.json)
- **`"icon": "third-party"`**: [Strict Compliance] In order to be correctly identified as a custom fragment on the Liferay DXP UI.
- **Explicit Definition of Paths**: Define all the following paths without omitting them.

**Implementation Example (`fragment.json`):**
```json
{
  "name": "ORG_PROJ_MyFragment",
  "type": "component",
  "icon": "third-party",
  "htmlPath": "index.html",
  "cssPath": "index.css",
  "jsPath": "index.js",
  "configurationPath": "configuration.json",
  "thumbnailPath": "thumbnail.png"
}
```

### 7.2 Auto-generation of Thumbnail Image (`thumbnail.png`) [Strict Compliance]
A thumbnail image to identify the content is required for each fragment.
- **Standard**: **320x160 px** PNG image.
- **Auto-generation Flow**: Upon creating or updating a fragment, the AI must always execute the **Python script (`scripts/generate_thumbnail.py`)** to physically generate the image.
- **Design**: The script draws a "random background color" and the "fragment name in the center".
- **Placement**: Place the generated image directly under each fragment directory with the name `thumbnail.png`.

2.  **Folder Name**: Make it match completely with the `name` in `fragment.json`.

## 10. Post-Implementation Validation (Workflow) [Strict Compliance]
After creating or updating the HTML of a fragment, always execute the following checking flow before completing the task to prevent recurrences.

1. **Check for HTML Inclusion**: Recheck the code to ensure that no HTML tags (`<i>`, `<span>`, `<img>`, etc.) are included inside tags with `data-lfr-editable`.
2. **Check for Editable Nesting**: Recheck to ensure that no other `data-lfr-editable` element exists inside a tag with `data-lfr-editable`.
3. **Execution of Corrections**: If you have generated code that violates the above, self-correct (wrap outside with a `div`, etc., or separate the editable attribute) before reporting to the user.
